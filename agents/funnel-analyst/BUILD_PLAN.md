---

## 1. The idea in one line

An AI agent that **automatically retrieves funnel/event data, analyzes the
conversion path stage-by-stage, and synthesizes a written report with
prioritised recommendations** — turning a raw CSV into a decision-ready brief
without anyone running the analysis by hand.

This is a textbook fit for the Data Analysis track, whose definition is:
*"Agent automatically retrieves data, analyzes, and synthesizes it into a report
or summary — without needing to do each step manually."*

## 2. Who it's for & the problem

| | |
|---|---|
| **User** | Growth / product / ops analysts who own a conversion funnel (signup, checkout, onboarding, loan application, etc.). |
| **Problem** | Funnel analysis is repetitive and manual: pull data, compute conversion at each step, find the biggest drop-off, slice by segment, then write it all up. It takes hours and is done inconsistently. |
| **Solution** | One command (or one chat message) → the agent computes the funnel deterministically and the LLM writes the executive summary + ranked recommendations. |
| **Value** | Minutes instead of hours; consistent, numbers-backed reporting; surfaces the highest-impact leak automatically. |

## 3. Architecture (retrieve → analyze → synthesize → report)

```
            ┌─────────────┐   ┌──────────────────┐   ┌────────────────────┐
 CSV / data │  RETRIEVE   │ → │     ANALYZE      │ → │     SYNTHESIZE     │ → report.md
  (synthetic)│  load+valid │   │ deterministic    │   │ LLM (Gemma/Qwen    │   (+ AI
            └─────────────┘   │ funnel metrics   │   │ via MaaS) narrative│    disclosure)
                              └──────────────────┘   └────────────────────┘
```

Key design decision: **the math is deterministic, the prose is the LLM's.**
Stage counts, conversion %, drop-off, velocity and segment tables are computed
in plain Python (auditable, never hallucinated). The LLM only *writes up* those
verified numbers into an executive summary and recommendations. This keeps the
report trustworthy while still showcasing agentic synthesis.

Components:

- `src/funnel.py` — deterministic metrics engine (stage-agnostic).
- `src/llm_client.py` — OpenAI-compatible MaaS client (configurable base URL,
  model, key). Works with public MaaS or self-hosted Gemma/Qwen instances.
- `src/synthesize.py` — builds the prompt from verified metrics, calls the LLM,
  formats the report. Degrades to a deterministic template if no key is set.
- `src/agent.py` — orchestrates the 4 steps; this is the callable the platform
  invokes.
- `run.py` — CLI / entrypoint.

## 4. Model choice (rulebook-compliant)

Primary: **MaaS self-hosted models** `gemma-4-31b-it` and `Qwen-3-27B`, accessed
through the GreenNode MaaS OpenAI-compatible API. This is free under the 10M VND
POC wallet and reflects the contest spirit. External models are allowed but
would have to be paid for and declared — we default to MaaS.

The client reads `LLM_BASE_URL`, `LLM_API_KEY`, `LLM_MODEL` from env so the same
code runs against the public MaaS endpoint or the team's own OpenClaw instance
URL (provided in the Training-day welcome email).

## 5. Data — strict compliance

The Tool Access Guideline is explicit: **only public, synthetic, or anonymized
data. No real customer data, no PII, no confidential/restricted internal data.**

We use **100% synthetic data** generated by `scripts/generate_sample_data.py`
(seeded, no real identifiers). The repo ships a synthetic SaaS-onboarding funnel
sample. If a judge points the agent at their own file, the README instructs that
it must also be synthetic/public/anonymized.

No internal tool access is required for the core demo, so no IT Helpdesk request
is needed. (If a future version reads from Office 365, the request goes to
helpdesk@vng.com.vn per Section 3 of the guideline, 8h SLA, personal resources
only.)

## 6. AgentBase deployment

- The agent's entrypoint (`agent.run(...)`) is wrapped so AgentBase can call it
  and get a report back — satisfying PASS criterion #1 (running on AgentBase,
  BTC can call it ≥1 time).
- Repo is pushed to GitHub (public or internal) and linked in the submission
  form — the agent must be running on AgentBase at submission time.
- Exact AgentBase wrapper/config follows the BTC sample template set (shared on
  Training day). The code is structured so the entrypoint is a single function,
  easy to bind to whatever AgentBase expects.

## 7. Mandatory AI disclosure

Per Rulebook §11.1, every agent must declare to end users that they are
interacting with AI. The report header and any chat response begin with a fixed
disclosure line (see `src/prompts.py` → `AI_DISCLOSURE`).

## 10. Risks & mitigations

| Risk | Mitigation |
|---|---|
| MaaS endpoint/model name differs from assumption | All LLM settings are env-configurable; confirm from welcome email. |
| Accidental use of non-synthetic data | Ship only synthetic data; README + COMPLIANCE.md state the rule prominently. |
| AgentBase wrapper format unknown until Training day | Entrypoint is one clean function; adapt the thin wrapper, not the logic. |
| LLM unavailable during demo | Deterministic fallback still produces a full report so the demo never breaks. |
